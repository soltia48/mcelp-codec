# デコーダのアーキテクチャとビットストリーム

## 1. フレームのタイミング

本コーデックは **8 kHz, モノラル** で動作します。1 つの圧縮フレームは
**18 バイト** で、復号すると **320 個の μ-law サンプル = 40 ms** の音声に展開されます。
内部的には、320 サンプルのフレームは階層的に分割されます。

```
frame   : 320 samples (40 ms)            ← 圧縮入出力の単位
 └─ block: 160 samples (20 ms)           ← ハーフフレーム、ポストフィルタ単位、LPC オーバライド単位
     └─ subframe: 80 samples (10 ms)     ← 励振 / 合成の単位
```

1 つのフレームは **2 ブロック** から成り、各ブロックは **2 サブフレーム** から
成る — 合計 4 サブフレームです。パイプラインのいくつかの段は「ブロック単位」
(LPC 解析、ピッチ状態) で構成されており、励振と合成は「サブフレーム単位」です。

## 2. トップレベルのデコードフロー

エントリーポイントは [src/lib.rs](../../src/lib.rs) の
[`Codec::decode_frame`](../../src/lib.rs) です。デコーダは 18 バイトの
フレームごとに以下のステージを順に実行します。

```
┌─────────────────────────────────────────────────────────────────────┐
│  18 バイトの入力                                                    │
│     │                                                               │
│     ▼                                                               │
│  is_reset_frame? ──yes──► 状態をリセットし、何も出力しない          │
│     │ no                                                            │
│     ▼                                                               │
│  canonicalize_payload          (bit 143 → 138 へ移動, 139..143 をクリア) │
│     │                                                               │
│     ▼                                                               │
│  ControlFrame::unpack          (14 フィールド + suppress フラグ)    │
│     │                                                               │
│     ▼                                                               │
│  ┌── ステージ 1: LSF 復号 ────────────────────────────────┐         │
│  │  LsfIndices.from_control                               │         │
│  │  combine_codebook (split-VQ 第 1 段 + 第 2 段)         │         │
│  │  enforce_min_gap × 2                                   │         │
│  │  predictive_combine (3 フレーム履歴)                   │         │
│  │  history.splice_new_vector                             │         │
│  │  stabilize_and_finalize (sort/clamp/min-gap)           │         │
│  │  lsf_to_lsp (cosine LUT)                               │         │
│  └────────────────────────────────────────────────────────┘         │
│     │                                                               │
│     ▼                                                               │
│  ┌── ステージ 2: LPC 解析 (3 回の response_shaper 走査) ─┐          │
│  │  パス A: lsp[0..10] | 0000 に対する安定性チェック     │          │
│  │           → 反射係数 k1..10                           │          │
│  │           → is_lpc_stable (block-0 オーバライドを判定) │          │
│  │  パス B: block-0 自己相関                             │          │
│  │           → sub0_lpc, sub1_lpc                        │          │
│  │  パス C: block-1 自己相関                             │          │
│  │           → sub2_lpc, sub3_lpc                        │          │
│  │  各パスは 35 セルのスクラッチバッファを引き継ぐ        │          │
│  └───────────────────────────────────────────────────────┘          │
│     │                                                               │
│     ▼                                                               │
│  ┌── ステージ 3: サブフレームごとのループ (i = 0..4) ───┐           │
│  │  decode_subframe_lag    (block 0/1 の絶対値 / 差分)   │          │
│  │  pitch_adaptive_codebook → v[80]                      │          │
│  │  fcb_dispatch_lag_class                               │          │
│  │  compute_pitch_enhance_gain                           │          │
│  │  fcb_short_path or fcb_main_path → c[80]              │          │
│  │  gain_orchestrate_codec → (g_p, g_c, history')        │          │
│  │  mix_excitation: e = 8·g_c·c + 4·g_p·v                │          │
│  │  lpc_synthesis_filter (1/A(z)) → s_hat[80]            │          │
│  │  past_excitation ← e                                  │          │
│  │  i が奇数なら: past_excitation をシフト, prev_lag を確定 │       │
│  └───────────────────────────────────────────────────────┘          │
│     │                                                               │
│     ▼                                                               │
│  postfilter_apply  × 2 ハーフフレーム                               │
│     │                                                               │
│     ▼                                                               │
│  linear_i16_to_ulaw × 320                                           │
│     │                                                               │
│     ▼                                                               │
│  320 バイトの μ-law 出力                                            │
└─────────────────────────────────────────────────────────────────────┘
```

このドキュメントの残りでは、ビットストリーム (ステージ 0) を扱います。
後続の 4 ステージはそれぞれ [lpc.md](lpc.md), [excitation.md](excitation.md),
[gain.md](gain.md), [synthesis.md](synthesis.md) のテーマです。

## 3. ビットストリーム

### 3.1 フレームレイアウト

各フレームは 18 バイト = 144 ビットで、MSB ファーストで読み出されます。
ペイロードは 14 個の連続フィールドに続いて小さな制御領域が並びます。

```
bit  0                        137  138    139            143
     │  14 フィールド, MSB ファースト │  S │ 未使用 (4 ビット) │  M │
     └──────────────────────────┴────┴─────────────────┴────┘
       138 ビットのペイロード        sup. (受信時にクリア)   フラグ送出元

S = bit 138 = "suppress フラグ" (正規化された格納先)
M = bit 143 = "frame flag source" (トランスポート層)
```

14 個のペイロードフィールドとそのビット幅 (詳細は [`FIELD_WIDTHS`][1]):

| Idx | 幅    | フィールド                              | 備考                   |
| --: | ----: | -------------------------------------- | ---------------------- |
| 0   | 8     | LSF mode (1 bit) + LSF stage-1 seed (7) | `mode = bit 7`        |
| 1   | 12    | LSF stage-2 上位 (6) + 下位 (6)        | split-VQ インデックス対 |
| 2   | 8     | ピッチ lag 絶対値, sub 0               |                        |
| 3   | 16    | FCB インデックス, sub 0                | 代数的符号語           |
| 4   | 7     | ゲイン位相ワード, sub 0                | ゲイン量子化インデックス |
| 5   | 5     | ピッチ lag 差分, sub 1                 | sub 0 からの相対値     |
| 6   | 16    | FCB インデックス, sub 1                |                        |
| 7   | 7     | ゲイン位相ワード, sub 1                |                        |
| 8   | 8     | ピッチ lag 絶対値, sub 2               |                        |
| 9   | 16    | FCB インデックス, sub 2                |                        |
| 10  | 7     | ゲイン位相ワード, sub 2                |                        |
| 11  | 5     | ピッチ lag 差分, sub 3                 | sub 2 からの相対値     |
| 12  | 16    | FCB インデックス, sub 3                |                        |
| 13  | 7     | ゲイン位相ワード, sub 3                |                        |

これらの幅は合計 138 になります。2 ブロック間の対称性は厳密で、F[2..7] は
sub 0/1 (block 0) を、F[8..13] は sub 2/3 (block 1) をパラメータ化します。
LSF (F[0..1]) はフレームに 1 度だけ送られ、両ブロックがローカルにそれを補間します。

### 3.2 suppress フラグと "flag-source" ビット

制御領域の 2 ビットがトランスポート層の状態を運びます。

- **bit 143** ("flag-source", `M`) はエンコーダのフレーマが生成したビットです。
- **bit 138** ("suppress", `S`) はデコーダのペイロードモデル内における
  そのフラグの正規化された格納先です。

受信側では (詳細は [`canonicalize_payload`][1] を参照) フレームを次のように
正規化します。

1. `[0..139)` のビットはそのままコピー
2. bit 143 → bit 138 へコピー
3. `[139..143)` のビットをクリア

正規化後、デコーダは bit 138 から `suppress` を読み取り、これは
`ControlFrame.suppress` として公開されます。現状のクレートは
非 suppress (`= 0`) パスを復号しています — フィールドはパースされて
呼び出し側に提供されますが、suppress=1 の代替ゲインパスはまだデコーダで
実行されません。

### 3.3 リセットフレーム

最後のバイトの下位ニブルが `0x8` (すなわち `(frame[17] & 0x0F) == 0x08`) で
終わるフレームは、インバンドの **リセットフレーム** として扱われます。
これは [`is_reset_frame`][1] によって検出されます。リセット時、デコーダは
すべての永続状態 (LSF 履歴、過去励振バッファ、LPC 合成履歴、ピッチ状態、
ゲイン履歴、ポストフィルタ状態など — [state.md](state.md) を参照) を
再初期化し、そのフレームでは音声を出力しません。

リセットは正規化の前に検出されるため、リセットコードはそれ自体が合法な
ペイロード範囲と共存します — これはペイロード値ではなく、フォーマットの
*マーカーとしてのフレーム* という規約です。

### 3.4 `.mcelp` テキストコンテナ

同梱の `.mcelp` ファイル (および `decode` バイナリ) は、シンプルで
ポータブルなテキストコンテナを使用します: 1 行 1 フレームで、各行は
18 バイトのフレームを 36 文字の 16 進数で表現したものです。CLI は
[`parse_frame_hex`][1] を使って変換します。これはコマンドライン実験のための
便宜的なものに過ぎず、コーデックライブラリ自体は生の 18 バイトスライスを
扱います。
